## 1. Цель MVP

Собрать первый рабочий релиз форума внутри Telegram Mini App, в котором пользователь может:

- авторизоваться через Telegram Mini App;
- увидеть список тем и их разделов;
- открыть секции/чаты внутри темы;
- просматривать, создавать, редактировать и удалять сообщения;
- работать с текстом, медиа, файлами и базовыми реакциями;
- работать с типами сообщений `POST`, `TASK`, `TASK_ASSIGNMENT`, `COMMENT`;
- редактировать описание раздела, если он владелец;
- управлять темами в отдельном модуле.

На 2-м этапе должна появиться независимая веб-версия с `email/password`, поэтому уже в MVP нельзя строить архитектуру только вокруг Telegram-данных.

---

## 2. Границы MVP

### Входит в MVP

- Telegram Mini App авторизация;
- только RU язык;
- fullscreen режим Mini App;
- safe area / отступы под системные и Telegram UI элементы;
- главный экран с поиском тем;
- экран темы с разделами и чатами;
- однотипные экраны чатов;
- CRUD сообщений;
- реакции;
- загрузка медиа и файлов;
- описание раздела;
- модуль управления темами.

### Не входит в MVP

- независимая веб-авторизация `email/password`;
- полноценная EN локализация;
- сложная модерация;
- роли и права сложнее базовых;
- push-уведомления;
- оффлайн-режим;
- продвинутые фильтры, сортировки и аналитика.

---

## 3. Базовые продуктовые правила

### Структура данных

Иерархия должна быть канонической:

`theme -> section -> chat -> message`

### Типы сообщений

- `POST` - обычная публикация/сообщение;
- `TASK` - задача;
- `TASK_ASSIGNMENT` - назначение/действие под задачей;
- `COMMENT` - комментарий под сообщением.

### Ограничения

- `COMMENT` создается только под сообщением, если у него `allow_comments = true`;
- `TASK_ASSIGNMENT` создается только под сообщением типа `TASK`;
- раздел "Описание раздела" не является чатом, там нет комментариев и реакций;
- большинство экранов чатов должны работать от конфигурации типа, а не от отдельных самописных страниц.

---

## 4. Что важно учесть по Telegram Mini App

По документации Telegram Mini Apps для MVP нужно сразу заложить:

- запуск в fullscreen mode;
- учет `safe area` и `content safe area`;
- адаптацию layout под перекрытие интерфейсом Telegram;
- серверную проверку `initDataRaw` от Telegram, а не доверие данным только с клиента.

Практически это значит:

- в корневом layout должен быть единый контейнер с верхним и нижним безопасными отступами;
- экран не должен сам вручную считать отступы в каждом месте;
- фронт отправляет `initDataRaw` на backend;
- backend валидирует Telegram init data и выдает внутреннюю сессию приложения;
- доменная модель пользователя хранится как внутренняя сущность, а Telegram - как провайдер входа.

## 5. Архитектура frontend (FSD)

Рекомендуемая структура:

```text
src/
  app/
  processes/
  pages/
  widgets/
  features/
  entities/
  shared/
```

### `shared`

Общее, без бизнес-смысла:

- api client;
- роутинг;
- ui-kit;
- config;
- lib;
- i18n;
- types;
- Telegram bridge / environment layer.

### `entities`

Базовые доменные сущности:

- `user`;
- `theme`;
- `section`;
- `chat`;
- `message`;
- `reaction`;
- `attachment`.

Внутри каждой сущности:

- `model`;
- `api`;
- `ui` при необходимости;
- `types`.

### `features`

Повторяемые пользовательские действия:

- `auth-by-telegram`;
- `search-themes`;
- `open-theme`;
- `view-chat-feed`;
- `create-message`;
- `edit-message`;
- `delete-message`;
- `react-to-message`;
- `upload-attachments`;
- `create-comment`;
- `create-task-assignment`;
- `edit-section-description`;
- `manage-themes`.

### `widgets`

Крупные блоки экранов:

- header приложения;
- список тем;
- список секций;
- лента чата;
- composer сообщения;
- карточка сообщения;
- панель фильтра/поиска.

### `pages`

Страницы:

- `home-page`;
- `theme-page`;
- `chat-page`;
- `section-description-page`;
- `theme-management-page`.

### Принцип

Сначала строим общие сущности и фичи, потом собираем из них страницы. Не делаем отдельную уникальную реализацию для каждого чата, если различие можно описать конфигом.

## 7. Что нужно определить на backend до начала экранов

До активной frontend-разработки нужно зафиксировать API-контракты.

### Обязательные backend-модули

- auth;
- users;
- themes;
- sections;
- chats;
- messages;
- attachments;
- reactions.

### Обязательные сценарии API

- вход через Telegram;
- получение текущего пользователя;
- список тем;
- поиск тем;
- список секций темы;
- список чатов темы;
- получение описания раздела;
- обновление описания раздела;
- список сообщений чата;
- создание сообщения;
- редактирование сообщения;
- удаление сообщения;
- создание комментария;
- создание `TASK_ASSIGNMENT`;
- постановка/снятие реакции;
- загрузка файла/медиа;
- CRUD тем для модуля управления.

### Важное правило API

Фронт должен получать не "экран 4 / экран 6 / экран 7", а универсальные данные:

- тип чата;
- доступные действия;
- флаги `allowComments`, `canEdit`, `canDelete`, `canPost`;
- метаданные для отображения.

Это позволит не плодить отдельную логику под каждый экран.

## 8. Порядок разработки

## Этап 0. Discovery и фиксация правил

Сначала:

1. Зафиксировать список сущностей, статусы и связи.
2. Утвердить, какие именно разделы являются чатами, а какие нет.
3. Утвердить права: кто может редактировать описание, темы, сообщения.
4. Описать JSON-контракты frontend/backend.
5. Проверить макеты и выписать все отличия между типами чатов.

Результат этапа:

- понятная модель данных;
- список API;
- список экранов;
- список отличий между `POST`, `TASK`, `TASK_ASSIGNMENT`, `COMMENT`.

## Этап 1. Технический фундамент приложения

Сначала делаем каркас проекта:

1. Инициализировать frontend-проект.
2. Подключить FSD-структуру.
3. Настроить роутинг.
4. Настроить state management и слой запросов к API.
5. Подготовить базовый UI-kit.
6. Подключить RU локаль.
7. Подготовить обработку ошибок, loader и empty states.

Результат этапа:

- приложение запускается;
- есть базовая архитектура;
- есть единый стиль и инфраструктура для экранов.

## Этап 2. Интеграция Telegram Mini App

Только после базового каркаса:

1. Подключить Telegram Mini App SDK/bridge.
2. Реализовать получение init data.
3. Реализовать fullscreen mode.
4. Реализовать safe area / content safe area отступы в корневом layout.
5. Добавить app shell с шапкой/отступами, чтобы UI Telegram не перекрывал контент.
6. Реализовать авторизацию через backend по Telegram init data.
7. Получать внутреннюю сессию приложения и текущего пользователя.

Результат этапа:

- пользователь может войти в приложение через Telegram;
- layout не ломается из-за Telegram UI;
- есть единая авторизация, которую потом можно расширить вторым провайдером.

## Этап 3. Базовые сущности и API-слой

После авторизации:

1. Создать `entities` для `user`, `theme`, `section`, `chat`, `message`.
2. Создать API-модули и типы.
3. Нормализовать DTO и маппинг в frontend-модели.
4. Подготовить общие хуки/сервисы загрузки данных.
5. Подготовить общие компоненты списков, карточек и статусов загрузки.

Результат этапа:

- фронт уже умеет работать с данными предметной области;
- следующие экраны собираются быстро и без дублирования.

## Этап 4. Главный экран

Дальше делаем главный пользовательский вход:

1. Экран списка тем.
2. Поиск тем по названию.
3. Загрузка существующих разделов по каждой теме.
4. Переход в тему.
5. Empty/error/loading состояния.

Результат этапа:

- пользователь видит форум и может начать навигацию по темам.

## Этап 5. Экран темы и список секций/чатов

После главного экрана:

1. Показать структуру выбранной темы.
2. Вывести секции и чаты.
3. Отдельно обрабатывать "Описание раздела" как не-чат.
4. Сделать конфигурацию отображения для разных видов секций.

Результат этапа:

- вся навигация внутри темы собрана;
- видно, куда попадает пользователь дальше.

## Этап 6. Универсальный движок чата

Это центральный этап MVP. Сначала делаем общую основу, а не каждый чат по отдельности.

1. Лента сообщений.
2. Карточка сообщения.
3. Composer для создания сообщения.
4. Редактирование сообщения.
5. Удаление сообщения.
6. Отображение вложений.
7. Базовые реакции.
8. Пагинация/дозагрузка.

Результат этапа:

- любой чат типа `POST` уже может работать на одной базе.

## Этап 7. Специальные сценарии сообщений

Когда общая лента готова:

1. Добавить `COMMENT` под сообщениями с `allowComments = true`.
2. Добавить `TASK`.
3. Добавить `TASK_ASSIGNMENT` под `TASK`.
4. Добавить различия в UI по типу сообщения, но на базе одного движка.

Результат этапа:

- поддержаны все основные типы данных MVP;
- логика остается общей и расширяемой.

## Этап 8. Раздел "Описание раздела"

После чатов:

1. Отдельный экран/режим просмотра.
2. Режим редактирования для владельца.
3. Сохранение описания.
4. Нет комментариев, реакций и чатовой логики.

Результат этапа:

- информационный раздел живет отдельно и не усложняет чатовый движок.

## Этап 9. Модуль управления темами

После пользовательского сценария:

1. Список тем.
2. Создание темы.
3. Редактирование темы.
4. Удаление темы.
5. Валидации и подтверждения опасных действий.

Результат этапа:

- администратор/владелец может управлять каталогом тем.

## Этап 10. Полировка MVP

В конце:

1. Проверить адаптивность в Telegram.
2. Проверить загрузку больших файлов и медиа.
3. Проверить состояния ошибок.
4. Проверить права доступа.
5. Добавить базовую аналитику и логирование ошибок.
6. Добавить smoke-тесты на критические сценарии.

Результат этапа:

- MVP готов к первому реальному использованию.

---

## 9. Приоритет реализации чатов

Чтобы быстрее получить рабочий результат, рекомендую такой порядок:

1. `Обсуждения`
2. `Копилка идей`
3. `Чат вопросов`
4. `Чат публикаций`
5. `Обмен опытом`
6. `Чат задач`
7. `Лаборатория экспериментов`
8. `ИКР`

Почему так:

- сначала поднимаем все простые `POST`-чаты;
- потом подключаем `TASK`-сценарии;
- потом добиваем особые разделы, если у них будут нюансы по макетам.

---

## 10. Что нужно подготовить сразу, чтобы не переделывать на 2-м этапе

Для будущей независимой веб-версии:

1. Сделать внутреннюю сущность сессии приложения.
2. Не использовать Telegram user id как единственный ключ всей системы.
3. Сделать слой `auth` с возможностью нескольких провайдеров.
4. Не зашивать Telegram-логику в доменные сущности `user`, `message`, `theme`.
5. Хранить `locale` и пользовательский профиль независимо от Telegram.

## 12. Definition of Done для MVP

MVP считается готовым, если:

- пользователь открывает Mini App в Telegram без проблем с layout;
- авторизация через Telegram работает стабильно;
- темы ищутся и отображаются;
- структура секций и чатов загружается с backend;
- в чатах можно просматривать и создавать сообщения;
- работают редактирование и удаление;
- работают комментарии там, где разрешены;
- работают `TASK` и `TASK_ASSIGNMENT`;
- работают реакции;
- работают медиа и файлы;
- описание раздела редактируется владельцем;
- темы можно просматривать/создавать/редактировать/удалять;
- нет критических UI-блокеров на основных сценариях.
